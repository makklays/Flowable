
Технический словарь терминов Flowable (EN / ES / RU)

1. Process Definition / Definición de proceso — Определение процесса (статический BPMN XML шаблон, загруженный в систему).
2. Process Instance / Instancia de proceso — Экземпляр процесса (конкретный запущенный в памяти/БД процесс).
3. Wait State / Estado de espera — Точка ожидания (шаг, на котором процесс останавливается и сохраняется в БД, ожидая внешнего действия, например, User Task).
4. Execution / Ejecución — Указатель выполнения (токен, который перемещается по схеме процесса и отслеживает текущий шаг).
5. Java Delegate / Delegado de Java — Джава-делегат (класс-обработчик бизнес-логики для Service Task).
6. Process Variable / Variable de proceso — Переменная процесса (данные, которые переносятся внутри процесса и сохраняются в БД).
7. Transient Variable / Variable transitoria — Транзитная переменная (переменная, которая живет только во время текущей транзакции и не пишется в БД).
8. Deployment / Despliegue — Развертывание (процесс загрузки BPMN-схем и ресурсов в базу данных движка).
9. Job Executor / Ejecutor de trabajos — Планировщик/выполнитель задач (компонент ядра, отвечающий за асинхронные шаги и таймеры).
10. Asynchronous Continuation / Continuación asíncrona — Асинхронное продолжение (разрыв транзакции на элементе схемы для выполнения его в фоновом потоке).
11. Exclusive Job / Trabajo exclusivo — Эксклюзивная задача (настройка, гарантирующая, что асинхронные задачи одного процесса не будут выполняться параллельно во избежание конфликтов).
12. Optimistic Locking Exception / Excepción de bloqueo optimista — Исключение оптимистичной блокировки (самая частая ошибка во Flowable, когда два потока пытаются одновременно обновить один и тот же процесс).
13. Boundary Event / Evento de límite (o adjunto) — Граничное событие (элемент, «прилепленный» к границе задачи для обработки ошибок, таймеров или сигналов).
14. Interrupting Event / Evento interruptivo — Прерывающий событие (граничное событие, которое при срабатывании отменяет саму задачу).
15. Non-interrupting Event / Evento no interruptivo — Непрерывающий событие (событие, которое запускает параллельную ветку, но оставляет основную задачу активной).
16. Call Activity / Actividad de llamada — Вызов подпроцесса (запуск внешнего самостоятельного процесса из текущего).
17. Multi-instance / Multi-instancia — Мульти-экземплярность (цикл в BPMN, создающий несколько копий одной задачи параллельно или последовательно).
18. History Level / Nivel de historial — Уровень логирования истории (настройка глубины сохранения данных: None, Audit, Activity, Full).
19. Decision Table / Tabla de decisión — Таблица решений (компонент DMN для реализации бизнес-правил в виде матриц).
20. Case Management (CMMN) / Gestión de casos — Управление кейсами (альтернативный BPMN движок для слабоструктурированных, динамических процессов).

1. Движки (Engines) внутри Flowable

BPMN Engine (Process Engine): Классические структурированные процессы (схемы со стрелочками от А до Я).
CMMN Engine (Case Engine): Гибкие процессы, где нет жесткой последовательности, а все зависит от возникающих событий (кейсов).
DMN Engine (Decision Engine): Движок правил. Нужен, чтобы не писать if-else в коде, а выносить логику (например, скоринг) в таблицы решений.
App, Form, Content Engines: Вспомогательные движки для UI-форм и работы с файлами.

2. Структура Базы Данных (Префиксы таблиц)

ACT_RE_* (Repository): Сюда попадает все, что ты деплоишь. Схемы процессов, картинки, DMN-таблицы. Таблицы: ACT_RE_PROCDEF (определения процессов), ACT_RE_DEPLOYMENT.
ACT_RU_* (Runtime): Здесь живут только активные данные. Как только процесс завершается, строки из этих таблиц удаляются. Это сделано для максимальной скорости работы. Таблицы: ACT_RU_EXECUTION (токены), ACT_RU_TASK (активные пользовательские задачи), ACT_RU_VARIABLE (текущие переменные).
ACT_HI_* (History): Сюда дублируются данные из Runtime для аналитики. Когда процесс умирает в ACT_RU_, вся его история остается здесь навсегда. Таблицы: ACT_HI_PROCINST, ACT_HI_ACTINST (шаги, которые прошел процесс).
ACT_GE_* (General): Общие системные данные. Самая важная таблица тут — ACT_GE_BYTEARRAY, где физически в виде BLOB/хранилища лежат твои XML-файлы процессов.

3. Транзакции: Важнейшая концепция

Запомни золотое правило Flowable, которое испаноязычные архитекторы называют «Transacciones ACID paso a paso» (Пошаговые ACID транзакции):
Flowable выполняет процесс в одном потоке Java и одной транзакции БД от одной точки ожидания (Wait State) до другой.

Пример:
  У тебя есть процесс: Start Event -> Service Task 1 (расчет) -> Service Task 2 (запись в БД) -> User Task (одобрение менеджером).
  1. Ты вызываешь runtimeService.startProcessInstanceByKey(...).
  2. Flowable открывает транзакцию в БД.
  3. В этом же потоке выполняет Service Task 1, затем сразу Service Task 2.
  4. Доходит до User Task. Это Wait State (точка ожидания).
  5. Движок записывает в ACT_RU_TASK новую задачу для менеджера и делает COMMIT в базу данных. Твой метод startProcessInstanceByKey завершает работу и возвращает управление.
Если в Service Task 2 упадет ошибка (Runtime Exception), то транзакция откатится (Rollback) до самого начала. В базе данных вообще не появится следов того, что этот процесс запускался!
